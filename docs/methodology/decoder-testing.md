# Testing strategy (PureCard decoder)

PureCard is a byte-level grammar/schema-constrained decoder for Legend Pure — a Rust library plus a thin, feature-gated PyO3 boundary (`docs/spec/architecture.md` §9), not a web server. Its correctness story is not the server pyramid in `docs/methodology/testing.md`; it is the **oracle-driven** strategy of `docs/spec/testing.md` §8, backed by two assets already committed to this workspace: `corpus/gold_queries.jsonl` (5,034 execution-verified gold Pure queries over 161 databases, §13.1) and `corpus/legend-stack/` (the pinned Legend engine 4.113.0 / SDLC 0.195.0, §14). This document is the layered-testing map for that strategy: what each layer proves, how its *valid inputs are mechanically generated*, where it lives, and which CI lane runs it.

The guarantee under test is asymmetric and silent (§8): a **soundness** bug masks a valid continuation and permanently blinds the model; a **completeness** bug lets it walk into a dead end the compiler rejects. Neither is visible without an oracle. That is why every §8 layer is built, and why the numeric floors are PROTECTED ratchets.

## The pyramid, mapped to a decoder

| Layer                                    | What it proves                                                                                           | Inputs (all mechanically derived)                                                 | Oracle                                                          | CI lane                                           |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- | --------------------------------------------------------------- | ------------------------------------------------- |
| **Unit**                                 | Local contract of one pure fn (PDA transition, mask intersection, N/T rule, `Schema::from_json`)         | Hand-authored golden pairs                                                        | The assertion itself                                            | Hermetic `just ci`                                |
| **Property** (proptest, §8.5)            | `accept_*` on any allowed token never panics and never dead-ends before an accepting state is reachable  | proptest-generated PDA states/stacks + a token the current mask allows            | Invariant                                                       | Hermetic `just ci`                                |
| **Mutation** (cargo-mutants)             | The suite *kills* logic mutants — no gate passes vacuously                                               | Mutants over `grammar/`, `mask/`, `schema/scope.rs`+`narrow.rs`                   | Killed/survived                                                 | Hermetic, separate job (`just test-mutation`)     |
| **Fuzz** (§8.4)                          | Random *accepting* walks all stay live; near-miss prefixes are rejected *exactly* (mask precision)       | Seeded random accepting PDA walks; gold prefixes truncated at offset *k*          | PDA liveness + (nightly) engine compile                         | Hermetic separate job (`just fuzz`) + engine half |
| **Differential-soundness** (§8.1)        | Never mask a valid continuation — the killer gate                                                        | All 5,034 `corpus/gold_queries.jsonl` records; L2 mode adds `corpus/schemas/*.md` | The gold corpus (valid by execution)                            | **Hermetic, every `just ci`**                     |
| **Differential-completeness** (§8.2/8.3) | No dead ends: constrained output parses *and* compiles; L2 output resolves with zero phantom/type errors | Constrained decoder output (accepting walks) + L2 counterfactuals                 | Live Legend engine, two-call `grammarToJson`→`lambdaReturnType` | **Engine-backed nightly** `just test-legend`      |
| **e2e** (M4)                             | The emergent "valid by construction" guarantee, live                                                     | PyO3 boundary + a real small model → constrained final-query span                 | Real Legend engine compiles the model's own output              | Engine-backed                                     |

## Why "e2e is not unit" here

For a thin server, an e2e request round-trip exercises little more than a unit, so "e2e ≈ unit" is a defensible shortcut. For PureCard that equivalence is a **misread**, and the reason is categorical, not a matter of degree.

- A **unit** test checks one pure Rust function against hand-picked inputs — `pda.rs` byte-transition tables, `mask.rs` set intersection, `scope.rs`/`narrow.rs` N/T rules. It proves a *local* contract in isolation, with no oracle beyond the golden value the author wrote down.
- **e2e** proves an *emergent* property that no unit can stand in for: that masking the logits of a real model, token by token, yields output that a **real external compiler** (Legend engine 4.113.0, §8.2/§14) accepts — through the real API surface (`DecoderSession` → PyO3, §9). The oracle is a ~1.7 GB multi-service Java stack that lives outside the process and outside the language. Nothing in a `#[cfg(test)]` module can substitute for "does Legend's type-checker return `TabularDataSet` or a compile error?"

So the decoder pyramid is stretched at the top: "unit" is a single `accept_token`/`allowed_mask` step; "e2e" is the full differential compile loop against a live oracle. The middle layers (property, fuzz, differential-soundness) are precisely the machinery that lets us *not* need the engine for the common case — they mechanically re-derive the oracle's verdict on the 5,034 cases we already have ground truth for, hermetically.

## The differential oracle: verdict on forking Legend

**Verdict: DO NOT fork. Reuse the two stock endpoints unchanged. Do not settle for "just compile" either.**

The steelman for forking is real and worth stating: fork the Legend monorepo, add a `/parseAst` endpoint, and diff PureCard's parse tree against Legend's for *structural agreement*, not just pass/fail — which would let us *localize* a soundness bug (which production is wrong) and catch queries that compile only by accident. It collapses on two facts:

1. **PureCard emits masks, not an AST** (§9.1). There is no PureCard parse tree to diff. For a *recognizer*, the invariant is language **membership**, not AST-equality: does the constrained byte-string **parse** and **compile**? AST-equality would only apply if PureCard deliberately built a parse tree — it does not, by design (§3.1, §4.4).
2. **`grammarToJson/lambda` already returns Legend's parsed AST** as protocol JSON (§14.2, Gate-0 findings). A hand-added "AST endpoint" would expose exactly what this stock endpoint already exposes — **zero new signal**.

Against that zero signal, forking is a real, recurring cost that the constitution weighs against directly: you inherit a ~1.7 GB multi-service build, you must track upstream 4.113.0, and you take on supply-chain and maintenance liability for a Java monorepo — all to re-expose an existing endpoint. That fails the "library before writing / don't gold-plate" bar and the §2 "cache or mirror, don't re-fetch" spirit.

The correct oracle is the **two-call stock differential** (§14.2), which is *stronger* than compile-only because it separates two membership questions the single compile call would blur:

- **(a) `POST /pure/v1/grammar/grammarToJson/lambda`** — input: the Pure lambda as raw **text**; output: the lambda as protocol JSON. This is the **parse** stage → **L1 parse-membership**. It needs no PMCD, so it is a cheap, schema-free syntactic gate: a malformed byte string errors here before any model loads. "Just compile" throws this cheap L1 signal away.
- **(b) `POST /pure/v1/compilation/lambdaReturnType`** — input: `{ lambda: <json from (a)>, model: <PMCD> }`; output: the return type (e.g. `TabularDataSet`) on success, a compile error on failure. This is the **type/name-resolution** stage → **L2 schema-consistency** (§8.3). Phantom identifiers and type mismatches surface here as compile errors; a returned type == completeness satisfied (§8.2).

So the differential is one-directional and membership-shaped: constrained walk → parses (a)? → compiles (b)? Reuse the two pinned finos images from `corpus/legend-stack/` unchanged.

## How valid test cases are generated

The decoder pyramid derives every input **mechanically** — no hand-picked query strings above the unit layer. That is what makes the gates un-gameable (§8.6) and the corpus, not intuition, the spec.

- **Unit — hand golden.** Small, explicit input→expected pairs for one function. Example: `mask.rs::BitMask::intersect(a, b)` asserted equal to `a ∩ b`; a `pda.rs` transition asserting that after the bytes `->groupBy(` the automaton *requires* `[` and dies on `)`.

- **Differential-soundness — the gold corpus (valid by execution).** Source = every `pure_text` in `corpus/gold_queries.jsonl` (5,034 records, 161 db_ids, arm split A=4,639 relational / C=395 class-nav, §13.1). Each string is *already execution-verified*, so it is valid by construction. At M0 (no tokenizer ships, §M0-spec) the harness drives the raw UTF-8 bytes one at a time and asserts the recognizer is never dead and is complete at end; at M1+ with a host vocab it tokenizes and asserts every actual next **token** stays in `allowed_mask()`. Concrete inputs, verbatim from the corpus:
  - arm A: `|spider::activity_1::Db->tableReference('default','Faculty')->tableToTDS()->groupBy([], agg('COUNT()', row: meta::pure::tds::TDSRow[1]|$row, y: meta::pure::tds::TDSRow[*]|$y->count()))`
  - arm C: `|spider::car_1::model::default::Countries.all()->filter(x|$x.continent == $x.fk0DefaultContinents.contId)` (the §6.4.2 worked example).

- **Fuzz / completeness — seeded random accepting walks (valid by construction).** Walk the compiled PDA from the start state; at each state pick uniformly among the tokens the mask currently allows; stop at an accepting state. The result is valid *by construction* because it never left the automaton's admissible set. Determinism via a seeded `StdRng` whose seed is drawn from `PROPTEST_SEED`/`RNG_SEED`/CI entropy each run and **printed on failure**, never hardcoded (constitution §7). Example: a synthesized `->filter(...)->project([...],[...])` pipeline never seen in the corpus, then POSTed to the engine → must compile (§8.2).

- **Mask-precision — near-miss prefixes (mechanically derived from gold).** Take any gold query, truncate at offset *k*: the gold byte/token *b* at *k* is by definition allowed. Enumerate the *other* candidates — at byte level, all 256 byte values except those the grammar admits at that state — and assert the mask **rejects** every one; where the grammar admits exactly one continuation, assert every alternative is masked. Slide *k* across each gold query and you get thousands of precision cases for free. Example: after `...->groupBy(`, inserting `)` where a `[` is mandatory must be rejected (this is the §5.2 empty-key `groupBy([]` shape's guard).

- **Property — proptest strategies.** Strategies generate `(grammar-state, token ∈ allowed_mask())` pairs directly (not query strings): a random reachable PDA state + stack, and a token sampled from that state's live mask. Invariant (§8.5): `accept_token` on that token never panics and never dead-ends before an accepting state is reachable. Seed env-injected, printed on shrink.

- **L2 — schema counterfactuals (mechanical single-identifier mutation).** Start from a gold query + its schema (`corpus/schemas/armC_ctx_car_1.md`, …), mutate exactly one identifier to make it illegal, and assert L2 masks it *and* the engine `lambdaReturnType` errors (§8.3). Two mutation families: **phantom** — rename `Faculty`→`Faclty`, or apply `Countries.all()->filter(x|$x.maker == 'Ford')` where `maker` is a CarMakers property, not a Countries one (§6.4.2 counterfactual); **type-mismatch** — feed a String-typed column into a numeric aggregate, or `boolProp > 3` (§6.6 T1/T2/T3). The *un-mutated* gold must still replay clean — that is the soundness half proving the mutation, not the grammar, is what the mask caught.

## Layers in detail

Crate module map (`docs/spec/architecture.md` §3.2): `grammar/{mod,pda,compiled}.rs`, `vocab.rs`, `mask.rs`, `recognizer.rs`, `schema/{mod,model,scope,narrow}.rs`, `session.rs`, `selfcheck.rs`, `error.rs`, `ffi.rs`. M0 kept only the dep-light core in `src/` (`lib.rs` + `vocab.rs`) and shipped the oracle harness under `tests/support/{error,recognizer,corpus,legend}.rs`, pulled into the integration-test binaries via `#[path]` (ADR-0003, per `specs/m0-oracle-harness.md`); M1 replaced the harness `StubDecoder` with the real byte-PDA behind the unchanged `ByteRecognizer` trait.

1. **Unit** — *what:* each pure fn's local contract. *Example fn:* `pda::transition_requires_open_bracket_after_groupby`; input: state after `->groupBy(`, expect `[` live / `)` dead. Also `vocab::from_byte_tokens` round-trip. *Location:* `#[cfg(test)]` in each `src/**` module. *Gate:* hermetic; counts toward coverage + mutation. (The oracle harness has its *own* unit tests of its pure helpers — e.g. `corpus::load_gold`'s malformed-line → `CorpusError::Json { line, .. }` — but they live in `#[cfg(test)]` modules under `tests/support/` (`tests/support/corpus.rs`), outside the `src/**` core and its coverage/mutation scope.)

2. **Property** — *what:* §8.5 no-panic / no-premature-dead-end. *Example fn:* `prop_accept_allowed_token_never_dead_ends`. *Location:* `tests/prop_pda.rs`. *Gate:* hermetic; seed from env, printed on shrink.

3. **Mutation** — *what:* the suite kills logic mutants over `grammar/`, `mask.rs`, `schema/scope.rs`+`narrow.rs`. *Example:* a mutant flipping a `<=` to `<` in `mask.rs`'s intersection must be caught by soundness or a unit. *Location:* `cargo-mutants` config; the feature-gated `ureq` POST is excluded by a **targeted per-function mutants-skip**, never a threshold drop. *Gate:* hermetic; **≥80% floor**, PROTECTED ratchet.

4. **Fuzz** — *what:* §8.4. Accepting-walk half feeds completeness; near-miss half proves precision hermetically. *Example fns:* `fuzz_accepting_walks_stay_live` (hermetic) and `fuzz_near_miss_prefix_rejected` (slide *k* over gold, assert non-gold bytes dead). *Location:* `tests/fuzz_walks.rs`, `tests/mask_precision.rs`; a `fuzz/` target for the OSS-fuzz-style corpus. *Gate:* hermetic (liveness + rejection); compile half on the engine lane. Seeds from CI entropy; a failing seed is **committed as a regression** while the sweep space stays live.

5. **Differential-soundness** — *what:* §8.1, the killer gate — never mask a valid continuation. *Example fn:* `replay_admits_entire_gold_corpus` (M0, byte-level, exists in `tests/soundness_replay.rs`); at M3 an L2 variant `replay_l2_admits_gold_for_schema` replays each gold under its matching `corpus/schemas/*.md`. *Concrete input:* the two gold strings above; corpus path resolved via `env!("CARGO_MANIFEST_DIR")` for detached-HEAD reproducibility (constitution §2). *Location:* `tests/soundness_replay.rs`, `tests/soundness_l2.rs`. *Gate:* **hermetic, every `just ci`**, full corpus (no sampling for the gate; sampling only for a fast pre-push hook). Asserts the exact `EXPECTED_GOLD_RECORDS` count (5,034) so a mutant that empties the loop or short-circuits the iterator reddens — the "100% soundness" claim has 5,034 real assertions behind it, not an assertion-free loop.

6. **Differential-completeness** — *what:* §8.2/8.3, no dead ends + L2 resolves. *Example fns:* `constrained_walks_all_compile` (accepting walks → `grammarToJson` parses → `lambdaReturnType` returns a type, target 100%), `l2_counterfactual_phantom_identifier_errors` (`Faculty`→`Faclty` must yield a compile error). *Location:* `tests/engine_compile_walks.rs`, `tests/l2_counterfactuals.rs`, both opening with `#![cfg(feature = "legend")]`; canned PMCD/protocol-JSON fixtures under `tests/fixtures/`. *Gate:* **engine nightly** `just test-legend`; §8.7 target — 100% constrained-generation compile + zero phantom/type errors on the OOS schemas.

7. **e2e** — *what:* the live emergent guarantee (M4). *Example fn:* `pyo3_real_model_final_span_compiles` — a Python loop builds a `Session` with the DB's schema at the final-query span, `&`-masks logits from a real small model each step, `accept_token`, stops at `is_complete()`+EOS; the produced lambda is POSTed to the live engine and must compile. *Location:* Python reference harness under the PyO3 boundary + `tests/e2e_pyo3.rs`. *Gate:* engine-backed.

## CI lanes & numeric floors

Two lanes, per §14.4, honest and no-skip:

- **Hermetic always-on (`just ci`).** What `xtask::ci()` actually runs on every PR: `cargo fmt --check`, two `cargo clippy` passes (default features, then `--all-features` so the feature-gated `legend` shim is compiled + linted), the `check-core-deplight` packaging gate, and the full workspace test build (`cargo test --workspace --all-targets`). That test build is where the hermetic §8 layers execute: unit + property (§8.5) + byte-level differential-soundness over all 5,034 gold (§8.1) + L2 soundness replay against `corpus/schemas/` (§8.1 L2-mode, from M3) + mask-precision + fuzz rejection half. Needs only the committed `corpus/` and (at M1+) the host tokenizer bytes — **zero docker, zero network**.
- **Hermetic but separate `just` targets / CI jobs (not part of `just ci`).** Coverage (`just coverage` → `cargo-llvm-cov`, ≥70% floor), mutation (`just test-mutation` → `cargo-mutants`, ≥80% floor once armed), and the standalone fuzz sweep (`just fuzz` → `cargo-fuzz`) are hermetic gates too, but they run as their **own** targets and dedicated CI jobs, not inside `xtask::ci()` — they are far slower, so folding them into the every-invocation `just ci` would make the inner loop unusable. They still gate the PR, just as separate jobs.
- **Engine-backed (`just test-legend`, nightly / on-demand).** Differential-completeness (§8.2) + L2 resolution (§8.3) + e2e (M4). Delegates to `cargo xtask test-legend`, which brings up `docker compose -f corpus/legend-stack/docker-compose.yml up -d`, runs `cargo nextest run --features legend` (each test health-waits the engine, `GET /api/server/v1/info`, per §14.1) on an **x86 runner** (images are amd64, ~1.7 GB total), then **always** tears the stack down.

**Numeric floors (PROTECTED, ratchet-up only, human-only lowering — constitution §3, §7):**

- **Coverage ≥ 70%** now (`cargo-llvm-cov`), on **default features only** — the `#[cfg(feature = "legend")]`-excluded shim cannot dilute the floor.
- **Mutation score ≥ 80%** initial (`cargo-mutants`) over the decoder logic. Mask computation and PDA transitions are dense and mutation-observable, so 80% is honest and achievable, then only rises.

**No-skip honesty mechanism.** The engine tests use a **file-level `#![cfg(feature = "legend")]`**, not `#[ignore]`. nextest never *collects* those tests unless `--features legend` is passed — the entire compilation unit is absent from the default build graph. There is no `#[ignore]`, no runtime `return`, no weakened assertion; the default gate's pass/fail is provably identical with or without a live engine (constitution §3's skip ban, satisfied honestly).

**Caching pinned engine images (constitution §2 — never a bare uncached fetch).** The two finos images (`finos/legend-engine-server-http-server:4.113.0`, `finos/legend-sdlc-server-fs:0.195.0`) are restored from `actions/cache` keyed on the version tag (a `docker save`/`docker load` tarball), so the nightly lane pulls from the registry on a cache **miss only** — an upstream rate-limit or 404 cannot redden the lane for reasons unrelated to the change.

## Anti-gaming

- **Non-hardcodable seeds.** The fuzz/property seed sweep draws seeds from CI entropy, never source. A failing seed is committed as a *pinned regression*, but the sweep space stays live so a grammar tuned to memorized outputs still faces fresh walks each run (constitution §7).
- **In-repo OOS partition (not yet a held-out anti-gaming suite).** The 3 **OOS** schemas segregated in the corpus (`oos_ctx_dog_kennels.md`, `oos_ctx_student_transcripts_tracking.md`, `oos_ctx_world_1.md`, §13.2) are a partition reserved for §8.7 generalization checks — a generator that overfits the visible pilot set (`armC_ctx_*`) should fail soundness/completeness on this OOS partition. **Honesty caveat:** these files are committed to this repository, so a generator with repo access can read them — they are **not** a generator-invisible held-out suite, and the kit claims no anti-gaming guarantee from them today. A true held-out suite (reviewer-controlled, injected from outside the repo at gate time) is **deferred and not yet enforced**; until it exists, the OOS partition is a generalization probe, not a tamper-proof gate.
- **The §8.6 corpus-derivation invariant *is* a test.** Any rule a gold query violates is *wrong*: the replay mechanically forbids masking a gold token. You cannot narrow the grammar to game mutation or precision numbers without reddening soundness — the corpus is a one-way ratchet on the grammar.
- **Assertion-backed "100%".** Asserting the exact `EXPECTED_GOLD_RECORDS` count (5,034) guarantees the 5,034 replay assertions actually ran; an emptied loop or vacuous gate fails, not passes.
- **CI recomputes floors independently.** Coverage and mutation scores are recomputed in CI; a self-lowered threshold is a CI failure and a reviewer red flag (constitution §7). Thresholds only ratchet tighter.

## Per-milestone rollout

Mapped to `docs/spec/overview.md` §10 milestones. Milestones M0–M5 are all **implemented** — the entries below record what each delivered. *Implemented* is not *everything-settled*, though: several scope and governance decisions remain open (see *Open decisions for the human* below, and the deferred L2 rules of §6.7 / `src/schema/mod.rs`). The code shipped; not every question the spec left to a human is closed.

- **M0 — oracle harness (shipped).** Unit tests (`error`/`vocab`/`recognizer`/`corpus`) + **byte-level** differential-soundness replay of all 5,034 gold through the harness `StubDecoder` (`tests/soundness_replay.rs`, exact `EXPECTED_GOLD_RECORDS` count) + engine POST skeleton behind `#[cfg(feature = "legend")]`. Floors armed: **coverage ≥ 70%** on default features. No token-level replay yet (no vocab shipped at M0).
- **M1 — L1 grammar.** Real byte-PDA replaces `StubDecoder` behind `ByteRecognizer`: byte-level soundness becomes a *real* gate (100% gold admits, zero premature dead-ends). Property tests (§8.5) + byte-level mask-precision + the accepting-walk generator come online. **Mutation floor ≥ 80%** armed over `grammar/`. Completeness lane starts: accepting walks → `grammarToJson`+`lambdaReturnType` nightly (§8.2), target 100% compile.
- **M2 — performance / mask cache.** Add a **metamorphic** gate: the cached per-state mask (§4.2) must equal the from-scratch recomputation (`mask_cache_equals_recompute`) — a cache bug is a soundness bug. Latency benchmark proves masking is off the critical path (≤ a few hundred µs/token). No new correctness *layer*, one new invariant. This is where token-level replay can begin **if** a host vocab fixture is supplied.
- **M3 — L2 schema overlay.** Scope/type tracker + narrowed terminals (§6). L2 differential-soundness replay against the 5 pilot `corpus/schemas/armC_ctx_*.md`; schema counterfactuals (phantom + type-mismatch); completeness L2 mode asserts **zero** phantom/type errors on the OOS-partition schemas (§8.3, §8.7). Mutation floor extends over `schema/scope.rs`+`narrow.rs`.
- **M4 — PyO3 boundary + e2e.** Layer 7: a Python loop drives a real small model, constrains only the final-query span, and the produced lambda compiles live. Token-level constraint is exercised end-to-end here (host supplies the vocab bytes, §9.3).
- **M5 — hardening.** Multi-seed fuzz sweep (CI-entropy seeds), tokenizer round-trip self-check at startup (tokenize→bytes→decoder over a gold sample, mitigating the §11 tokenizer-exactness risk), incomplete-generation handling, final benchmarks. Floors ratchet up as coverage/mutation headroom is proven. (A reviewer-controlled held-out slice remains **deferred, not yet enforced** — see *Anti-gaming* above; the in-repo OOS partition is a generalization probe, not a tamper-proof gate.)

## Open decisions for the human

These are the questions the spec deferred to a human. With M0–M5 now implemented, most have a disposition (recorded inline below); a couple remain genuinely open. This list is kept as the audit trail — a resolved item is annotated, not deleted.

1. **Token-level soundness needs a committed vocab fixture — or stays byte-level until M4.** The repo ships *no* tokenizer/vocab (filesystem-verified, M0 spec). Byte-level replay proves the PDA admits every gold byte stream; it does **not** prove token-id soundness (a byte-admissible stream can still mask a specific token id at a BPE boundary — M0-spec R1). Decision: commit a small frozen vocab fixture to make token-level soundness a hermetic M2 gate, or defer token-level assertions to the M4 engine/e2e lane. Recommendation: commit a minimal fixture at M2 so the real §8.1 guarantee is gated hermetically, not only live. **Disposition (M5):** resolved by *deferral* — no hermetic vocab fixture was committed, so the always-on replay gate stays byte-level; token-id soundness is instead exercised by the M5 tokenizer round-trip self-check at startup and the M4 e2e lane live, not in hermetic replay.
2. **Mutation floor scope and starting value.** 80% over `grammar/`+`mask/`+`schema/` is proposed as honest-and-achievable. Confirm the floor value and the exact module set before M1 arms it (PROTECTED once set — humans only lower it). **Disposition:** resolved — the **≥80%** floor is armed over `grammar/`+`mask.rs` at M1 and extended over `schema/scope.rs`+`narrow.rs` at M3 (see *Mutation* above), a PROTECTED ratchet.
3. **Held-out gold slice size.** The 3 OOS schemas are a natural held-out schema set; the *gold-query* held-out slice size (e.g. withhold N db_ids entirely from grammar authoring) is a reviewer call. Recommendation: withhold whole db_ids, not random lines, so the held-out set tests generalization across schemas, not memorization within one. **Disposition:** still **open** — the in-repo OOS partition is a generalization *probe*, not a tamper-proof held-out suite; a reviewer-controlled, generator-invisible held-out suite is deferred and not yet enforced (see *Anti-gaming* above).
4. **Engine nightly runner.** Confirm an x86 CI runner is available for the ~1.7 GB amd64 stack; otherwise the completeness lane stays local/on-demand (`just test-legend`) and only the hermetic gate is enforced pre-merge. Either way the always-on soundness gate is unaffected. **Disposition:** resolved — the engine-backed lane runs on an x86 runner, bringing the amd64 stack up via `docker compose` and tearing it down after (see *CI lanes* above); the completeness lane is nightly/on-demand, the hermetic soundness gate every PR.
5. **`map`'s argument grammar (§11 open L1 gap).** `map` (~31 gold) was originally "treat as `fn`" but takes a lambda `fnArgs` cannot accept — a completeness gap that risked a small number of `map` gold queries failing byte-level soundness. **Disposition (M1):** resolved by *folding* the `map`-lambda production into the grammar (constitution §6, fold-vs-branch); all 5,034 gold queries — the `map` ones included — now admit at byte level, so the §8.1 soundness gate holds at the full corpus count.
